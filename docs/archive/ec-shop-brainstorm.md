企画としては「ECモック」より、AIで迷いを減らす売り場デモにした方が営業で刺さります。最新仕様に依存する部分は最小にして、Cloudflareは静的配信、SupabaseはAIゲートウェイに限定する前提で組みます。

思考時間: 7s
1. 結論

この企画なら、「美容・ギフト系EC」 が一番強いです。

理由は、AIコンシェルジュと検索リランキングの価値が見えやすいからです。

テーマ:
AIコンシェルジュ付き セレクト美容・ギフトショップ

対象:
20代後半〜40代向け
美容品 / 雑貨 / ギフト / セルフケア商品

狙い:
「商品が多すぎて選べない」をAIで解決するデモ

技術構成はこれで十分です。

React JS
Cloudflare Pages
Supabase Edge Functions
OpenAI API
DBなし
ログインなし
商品データは固定JS/JSON
注文履歴はlocalStorage

OpenAI APIはブラウザ直呼びではなく、Supabase Edge Functions側から呼ぶ形が安全です。Supabase公式もOpenAI APIはサーバーサイド利用を前提とし、Edge Functions経由の利用例を出しています。
Cloudflare PagesはReact/Viteの静的デプロイに対応しており、ビルド出力は通常 dist を指定できます。

2. 理由

Amazon風の汎用ECにすると、機能が広すぎてデモの焦点がぼけます。

営業用で刺さるのは、これです。

普通のEC:
商品を並べる

AI付きEC:
迷っている客に、理由つきで選んであげる

特に美容・ギフト系は、顧客の悩みが自然言語になりやすいです。

「彼女へのプレゼント」
「肌が弱い」
「予算5,000円」
「失敗しにくいもの」
「忙しい人向け」
「初心者でも使いやすい」
「高見えするギフト」

つまり、検索キーワードではなく相談文が発生する商材です。

ここでOpenAI APIを使う価値が出ます。

5. 検索リランキング設計

せっかくOpenAI APIを入れるなら、検索リランキングだけは実装する価値があります。

ただし、ベクトルDBやSupabase DBは不要です。

最小リランキング

流れはこれです。

1. React側で通常検索して候補を10〜15件に絞る
2. 候補商品だけSupabase Edge Functionへ送る
3. OpenAIに「相談文に合う順に並べ替えて」と依頼
4. productIdの順序だけ返す
5. React側で並べ替えて表示

これならDB不要です。

重要なのは、OpenAIに全商品を投げないことです。
商品数が少ないモックなら全件でも動きますが、本番化を意識するなら候補だけ渡す方が綺麗です。

Edge Function名
recommend-products

リクエスト：

{
  "query": "予算5000円で彼女へのプレゼント",
  "candidates": [
    {
      "id": "gift-001",
      "name": "リラックスアロマセット",
      "price": 4800,
      "tags": ["ギフト", "香り", "リラックス"]
    }
  ]
}

レスポンス：

{
  "message": "予算5,000円以内で、ギフト感がある商品を優先しました。",
  "rankedProductIds": ["gift-001", "skincare-002", "relax-003"],
  "reasons": {
    "gift-001": "価格が合い、プレゼント感が強いです。",
    "skincare-002": "実用性が高く、失敗しにくいです。"
  }
}

この形式ならReact側が扱いやすいです。

6. AIコンシェルジュ設計

コンシェルジュは、検索とは分けた方が良いです。

検索リランキング:
入力に対して商品一覧を並べ替える

コンシェルジュ:
会話形式で悩みを整理して、商品を提案する

画面上はこう分けます。

商品一覧:
AIで並び替え

コンシェルジュ:
相談して選ぶ

コンシェルジュの返答は、商品提案だけではなく、質問返しを入れるとAI感が出ます。

おすすめ3件を出したあと、
「香りがある商品は大丈夫ですか？」
「相手の年代は？」
「実用重視ですか、見た目重視ですか？」

ただし、営業デモでは会話を深くしすぎない方が良いです。
2ターン以内で商品提案まで行くべきです。

7. 破綻条件

この企画が破綻する条件はこれです。

1. AIが毎回違う回答をしてデモが不安定になる
2. 商品データが薄くてAIの提案理由が弱い
3. 画面がECそのものに寄りすぎてAI価値が見えない
4. 注文・履歴を作り込みすぎて本質からズレる
5. Supabase Functionが無制限に叩けてコスト事故になる

特に注意は5です。

Supabase Edge Functionsはサーバー側関数として使える一方、Secrets管理やJWT検証などの扱いを誤ると公開デモで叩かれ続けます。SupabaseはEdge Functionsのシークレット管理を提供しており、ブラウザに出してよいキーと出してはいけないキーを分けています。

営業デモなら、最低限これを入れます。

入力文字数制限
1回の候補商品数制限
固定カテゴリ外の質問は断る
エラー時フォールバック
公開URLを広く拡散しない
必要なら簡易デモコード
