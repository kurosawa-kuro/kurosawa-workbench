# アニメーション対策（Interaction Design Rebuild）

## Goal

Pico.css を最低限の根本スタイルに留め、Motion / GSAP / View Transitions を前面に出して、営業デモとして「AI が購買体験を動かしている」印象を作る。

## Background

現状は機能・商品データ・画像素材・AI 接続・購入フロー・管理画面の土台は揃っている。一方で、基本的なシステムとしてのデザイン密度が弱く、画面遷移や AI 提案の体験が静的に見える。

色の微調整より先に、以下を整える必要がある。

- 画面ごとのレイアウトルール
- 商品カードと推薦カードの情報設計
- AI 相談から提案までの体験演出
- 商品一覧の検索・リランキング時の変化
- カート追加、checkout、complete の流れ
- Admin dashboard の KPI / ランキング表示

## Value

バックエンドが薄いデモアプリなので、フロントエンド表現に寄せてもバグリスクが比較的小さい。アニメーションを適切に入れることで、単なる EC デモではなく「AI 導入後の購買体験」として見せられる。

## Design Policy

### Pico.css

- 残してよいが、主役にしない。
- フォーム、button、基本 typography など最低限の土台として扱う。
- カード、ページレイアウト、商品一覧、AI 提案、Admin dashboard は自前 UI として設計する。

### Motion

アプリ全体の標準アニメーション担当。

- ページ入場
- 商品カードの stagger reveal
- フィルター変更時のカード更新
- AI 回答カードの出現
- カート追加時のフィードバック
- checkout step transition
- Admin KPI / ranking の reveal

### GSAP

見せ場専用。全画面に濫用しない。

- トップヒーロー
- 商品画像や推薦カードの印象的な動き
- AI が候補をスキャンしているような演出
- Admin dashboard 初回ロード演出

### View Transitions

画面間のつながりを作る。

- 商品一覧 → 商品詳細
- コンシェルジュ → 商品詳細
- カート → checkout → complete
- SPA でも「画面が切り替わった」ではなく「体験がつながった」印象にする。

## Scope

- `motion` の導入
- 必要なら `gsap` / `@gsap/react` の導入
- View Transition API の導入可否検証
- 共通 animation primitives の追加
  - page transition
  - stagger list
  - animated card
  - animated count / KPI
  - loading / thinking state
- トップ画面のヒーロー演出
- 商品一覧のカード reveal / AI リランキング時の並び替え演出
- 商品詳細への遷移演出
- コンシェルジュの相談送信 → thinking → 推薦カード出現
- カート追加時の feedback
- checkout step の transition
- Admin dashboard の KPI / ranking reveal
- 既存 E2E の文言・導線更新

## Non-scope

- AI API の仕様変更
- Supabase Edge Functions の大幅変更
- 商品データの追加
- 色・ブランドトーンの最終調整
- 本格的なデザインシステム構築
- Three.js など 3D 表現の導入

## Implementation Notes

- まず Motion を主軸にする。
- GSAP はトップや Admin など「見せ場」に限定する。
- View Transitions はブラウザ対応を確認し、未対応環境では通常遷移にフォールバックする。
- `prefers-reduced-motion` を尊重する。
- アニメーションは UI の理解を助けるために使い、操作を遅くしない。
- ページ遷移や一覧更新で layout shift が起きないよう、カードや画像の寸法を固定する。
- 商品画像のないカードは、アニメーション前にまず見た目の弱さが出るため、プレースホルダーの扱いも見直す。

## Suggested Phases

### Phase 1: Foundation ✅ 完了

- [x] Motion を導入する。（`npm install motion`）
- [x] 共通の animation primitives を作る。（`src/lib/motion.js` — staggerGrid / staggerFast / cardReveal / slideLeft / slideRight / fadeUp / dealCard）
- [x] `prefers-reduced-motion` 対応を入れる。（CSS `@media` + `MotionConfig reducedMotion="user"` in App.jsx）
- [x] ページ共通の入場アニメーションを追加する。（Top ヒーロー: 左右スライドイン、各ページ: stagger reveal）

### Phase 2: EC Experience ✅ 完了

- [x] 商品カードを再設計する。（4 列グリッド、インラインカートボタン、カテゴリ別グラデーション）
- [x] 商品一覧の stagger reveal を追加する。（`motion.section` + `motion.article` + `staggerGrid`）
- [x] AI リランキング時にカードが滑らかに並び替わるようにする。（`layout` prop — カードが元位置から新位置へ物理移動）
- [x] 商品一覧 → 詳細の遷移を改善する。（`startViewTransition` + `TransitionLink` 導入済み）

### Phase 3: AI Concierge Experience ✅ 完了

- [x] 初期状態と回答後状態を分ける。（`hasSubmitted` state で empty → result を切り替え）
- [x] 相談送信後に thinking state を表示する。（スケルトンカード×3 shimmer + scan bar + Sparkles 回転アイコン）
- [x] 推薦カードを 1 枚ずつ理由付きで出す。（spring "deal" animation — 160ms stagger、左から弾道飛来）
- [x] カート追加時に視覚的 feedback を出す。（`cart-fly` カスタムイベント → 商品画像が放物線でカートへ飛翔 → カートアイコンがバウンス）

### Phase 4: Purchase / Admin ✅ 完了

- [x] カート → checkout → complete の step transition を追加する。（View Transition API で画面遷移をつなぐ）
- [x] Admin dashboard の KPI を animated count で表示する。（`CountUp` コンポーネント — `useSpring` で 0 → 実値へカウントアップ）
- [x] ランキングと改善提案を reveal する。（`motion.ol` + `motion.li` stagger / 改善提案は `whileInView` scroll-triggered）

### Phase 5: Polish ✅ 完了（GSAP は見送り判断）

- [x] GSAP を必要箇所だけ導入する。→ **Motion で十分カバーできたため見送り。** GSAP 追加を要するほどの複雑な showpiece は不要と判断。
- [x] View Transitions を導入または見送り判断する。→ **導入済み。** `src/lib/viewTransition.js` + `src/components/TransitionLink.jsx`。未対応環境はフォールバック。
- [x] E2E を更新する。（h1 テキスト / ボタンラベル / フォームラベルを現行 UI に追随済み、15/15 pass）
- [x] 最後に色、余白、フォントサイズを微調整する。（step-badge 可読性修正、モバイル対応、header-nav-admin 追加など）

## Stop / Ask Owner If

- GSAP を全面採用したくなる場合
- View Transitions のブラウザ対応を理由に UX が分岐しすぎる場合
- アニメーション追加が操作速度や可読性を下げる場合
- 商品画像素材の不足がデザイン品質を大きく制限する場合
