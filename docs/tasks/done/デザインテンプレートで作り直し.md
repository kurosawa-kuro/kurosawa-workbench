# フロントエンド再構築・デザイン統一依頼

## 目的

現在のフロントエンドは、機能追加や個別修正を継ぎ足してきた結果、以下の問題が多数発生しています。

* ページごとにレイアウトや余白が異なる
* コンポーネント間で高さ・幅・配置が揃っていない
* フォーム、ボタン、テーブル、カードなどのデザインに一貫性がない
* CSSの局所的な上書きが増え、修正によって別の箇所が崩れやすい
* レスポンシブ表示に不自然な箇所がある
* 既存コードを部分修正するだけでは、歪みを吸収しきれない

そのため、既存画面を継ぎ接ぎで修正するのではなく、完成度の高い既存フロントエンドをコピーして新しい土台とし、現在の機能を移植する形で再構築してください。

## 土台の候補

### 第一候補：Pico CSSベース

```text
/home/ubuntu/repos/portfolio/admin-pico
```

現在の技術構成に近く、移行コストを抑えられる可能性があります。

### 第二候補：Tailwind CSSベース

```text
/home/ubuntu/repos/private-ops/starter-kit/assets/optional/admin/react
```

デザイン品質とコンポーネント完成度は、こちらが最も高いと考えています。

今回に限り、デザインと保守性の改善効果が明確であれば、Pico CSSからTailwind CSSへの切り替えを許可します。

## 依頼内容

最初に、以下を調査してください。

1. 現在のフロントエンド構成
2. デザインが崩れている箇所と、その原因
3. 共通レイアウト、共通コンポーネント、CSSの依存関係
4. `admin-pico`を土台にした場合の移植難易度
5. Tailwind版React Adminを土台にした場合の移植難易度
6. 既存機能を維持したまま再構築できる範囲
7. Picoを維持する場合とTailwindへ切り替える場合の、品質・工数・保守性の差

そのうえで、より適切な土台を選定し、実際にフロントエンドを再構築してください。

## 基本方針

単なる見た目の模倣ではなく、選定した候補の以下の構造を土台として利用してください。

* アプリ全体のレイアウト
* ヘッダー
* サイドバー
* ナビゲーション
* ページコンテナ
* カード
* フォーム
* ボタン
* テーブル
* モーダル
* アラート・通知
* ローディング・空状態・エラー状態
* レスポンシブ設計
* デザイントークン
* コンポーネントのサイズ・余白・整列ルール

現在の画面をそのまま上からCSSで矯正するのではなく、新しい共通レイアウトと共通コンポーネントへ既存機能を移植してください。

## 特に重視すること

### 1. コンポーネントのズレを構造的に解消する

ページごとの個別調整ではなく、共通コンポーネント側で以下を統一してください。

* 高さ
* 幅
* 余白
* 枠線
* 角丸
* フォントサイズ
* ラベル位置
* 入力欄とボタンの配置
* テーブルの列・行間隔
* カード内部の配置
* モバイル時の並び方

### 2. 既存機能を壊さない

今回の主目的はフロントエンドの再構築です。

以下は原則として維持してください。

* API呼び出し
* データ取得処理
* 状態管理
* バリデーション
* 画面遷移
* ユーザー操作フロー
* 既存の業務ロジック

不要なバックエンド変更や、機能仕様の変更は行わないでください。

### 3. 継ぎ接ぎを増やさない

以下の対応は避けてください。

* ページ単位の大量なCSS上書き
* `!important`による強制調整
* 同じ用途のコンポーネントの重複作成
* 根拠のない固定幅・固定高さ
* 一時対応のためだけのラッパー追加
* 新旧デザインシステムの長期的な混在
* TailwindとPicoの中途半端な併用

移行期間中に一時的な混在が必要な場合も、最終的にどちらへ統一するかを明確にしてください。

## 土台の選定基準

次の優先順位で判断してください。

1. 最終的なデザイン品質
2. コンポーネント間の一貫性
3. レスポンシブ対応
4. 保守性
5. 既存機能の移植しやすさ
6. 今後の機能追加のしやすさ
7. 移行工数

単に移行工数が少ないという理由だけでPicoを選ばず、Tailwindへ切り替えることで根本的な改善が得られる場合はTailwindを選択してください。

反対に、`admin-pico`を土台にしても十分な品質と統一性を実現できる場合は、不要な技術切り替えは避けてください。

## 進め方

以下の順序で進めてください。

1. 現状のフロントエンドを調査する
2. 2つの候補を比較する
3. 採用する土台を決定する
4. 採用理由と不採用理由を簡潔に説明する
5. 共通レイアウトとデザインシステムを先に整備する
6. 共通コンポーネントを作成・移植する
7. 各ページを新しい土台へ移植する
8. 不要になった旧CSS・旧コンポーネントを削除する
9. PC・タブレット・モバイル表示を確認する
10. lint、型チェック、テスト、ビルドを実行する
11. 残存するデザイン上の不整合を確認する

## 完了条件

以下を満たした状態を完了とします。

* 全ページが同一のレイアウトルールで表示される
* 同じ用途のコンポーネントが共通化されている
* フォーム、ボタン、カード、テーブルの位置やサイズが揃っている
* PCとモバイルの両方で大きな表示崩れがない
* 既存機能と操作フローが維持されている
* 局所的なCSS上書きが大幅に削減されている
* 旧デザインと新デザインが不自然に混在していない
* lint、型チェック、テスト、ビルドが成功する
* 今後の画面追加でも同じ共通コンポーネントを利用できる

## 最終報告

作業完了後、以下を報告してください。

* 採用した土台
* その採用理由
* PicoまたはTailwindを不採用にした理由
* 再構築したレイアウトと共通コンポーネント
* 削除または置き換えた旧実装
* 既存機能への影響
* 実行した検証
* 残っている課題
* 今後、再びデザインが継ぎ接ぎ化しないためのルール

細かなデザイン調整を個別に積み重ねるのではなく、今回の作業でフロントエンドの土台そのものを正常化することを最優先にしてください。
## 実施結果（2026-07-13）

### 採用判断

- 採用: `/home/ubuntu/repos/portfolio/admin-pico`
- 理由: 既存の React / Pico CSS / API 処理を維持したまま、ダーク固定トークン、サイドバー型シェル、カード、フォーム、レスポンシブ規則を移植できるため。
- Tailwind 版を不採用にした理由: 今回必要な品質と統一性は admin-pico で実現でき、React コンポーネントとスタイル記法を全面移行する追加コストに見合う差がないため。

### 再構築内容

- `AppShell`: デスクトップの固定サイドバー、モバイルのオフキャンバスメニュー、共通フッター。
- `ui.jsx`: `PageHeader`、`SectionHeader`、`Card`、`Badge`。
- CSS: `admin-theme.css`、`admin-layout.css`、`admin-components.css`、`pages.css`、`admin-responsive.css` に責務分割。
- 全6ページを共通シェルと共通コンポーネントへ移植。
- AI相談フォーム、ローディング、空状態、エラー状態、結果表示を同じトークンとカード規則へ統一。

### 削除・置換

- 旧 `Header.jsx` / `Footer.jsx` を `AppShell.jsx` に置換。
- 旧 `App.css` と単一巨大CSSを廃止し、Pico CSS + admin-pico 派生スタイルへ置換。
- 未使用の Vite 初期アセットと View Transition helper を削除。

### 既存機能への影響

API 呼び出し、フォーム状態、バリデーション、localStorage の相談履歴、画面遷移、AI結果のコピー / mailto 導線は変更していない。

### 検証

- `npm run lint`: 成功
- `npm run build`: 成功
- `npx playwright test`: 12件成功
- desktop 1440px / tablet 900px / mobile 390px で全6ルートの横 overflow がないことを自動確認。
- デスクトップとモバイルでトップ、サービス、経歴、AI相談を目視確認。
- モバイルメニューの開閉と遷移後の自動クローズを確認。

### 残課題

- Vite の 500kB 超 JS チャンク警告は残る。既存の Motion / Supabase 依存を含む性能改善課題であり、今回のデザイン再構築範囲外。
- DeepSeek / Supabase の実接続は外部環境依存のため、今回の自動検証には含めていない。

### 継ぎ接ぎ化を防ぐルール

1. 新規ページは必ず `AppShell`、`PageHeader`、`Card` など既存プリミティブを使う。
2. 色・高さ・角丸・余白は `admin-theme.css` のトークンを使い、ページ内へ直値を増やさない。
3. 共通部品の変更は `admin-components.css`、ページ固有表現は `pages.css` に限定する。
4. `!important`、Pico と Tailwind の併用、用途が重複するコンポーネント追加を禁止する。
5. 変更後は3 viewport の overflow テストと全6ルートのスモークテストを通す。
