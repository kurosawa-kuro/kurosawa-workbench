# アイデア: スマホデモで魅力が伝わる機能追加

- 日付: 2026-07-06
- 種別: idea（仕様・設計・実装方針はスコープ外。着手時に別タスクへ切り出す）
- 相談の前提: このアプリは EC や収益目的ではなく、**スマホで見せながら「AI を活用した Web アプリを作れる人」だと短時間で伝えるための営業・コミュニケーションツール**である。

## 評価軸（今回の相談観点）

1. スマホで見せたときに短時間で魅力が伝わるか
2. AI を活用していることが自然に伝わるか
3. 会話のきっかけになるか
4. 「AI 関連の仕事をしている人が作ったアプリ」として印象に残るか

## 現状の強みと、デモ観点での弱点

強み:

- AI 相談が「ただのチャット」ではなく fit 判定・スコープ・問い合わせ文への**構造化された回答**になっており、既に差別化されている
- パイプライン表示（案件入力 → AI 照合 → fit 判定）やスケルトン演出など、「AI が動いている感」の下地がある

デモ観点での弱点:

- **見せ場に到達するまでに入力が必要**。スマホを見せながらの立ち話で、500 文字のフォーム入力は成立しない
- 回答が一括表示のため、待ち時間が「無言の数秒」になる。デモではこの数秒が一番気まずい
- AI 体験が `/ai-consult` の 1 箇所に閉じており、他ページは「よくできた静的サイト」に見える
- 相手（見せる相手）は登場しない。話題が常に「黒澤の案件相談」なので、相手側の会話が生まれにくい

## 優先アイデア

優先順は「デモの成功率を上げるもの → 会話を生むもの → 記憶に残すもの」。

### 1. ワンタップデモ + ストリーミング表示（デモの成功率を上げる）

相談例チップのタップだけで送信まで完了する「30 秒デモ」導線と、回答をトークン単位で流し込むストリーミング表示。

- **なぜ効くか**: ChatGPT で世間に刷り込まれた「AI が考えながら書いている」画は、ストリーミングそのもの。これが出た瞬間に「AI が動いている」ことが説明ゼロで伝わる（観点 1・2）
- デモの最大の敵は「入力の手間」と「無言の待ち時間」。両方を同時に潰す
- 補助案: 通信不安・API 障害・利用回数制限に備えた**オフライン再生モード**（実際の回答例をストリーミング風に再生する録画的デモ）。営業の現場で「今日はつながらない」を絶対に起こさないための保険

### 2. 音声対応（AI が「身近」になる瞬間を作る）★ 主軸

案件概要をスマホに向かって話すと、そのまま AI 判定まで進む。

- **なぜ効くか**: 「スマホに話しかけたら AI が答えた」は、非エンジニアに一番刺さる体験。技術説明が一切いらない（観点 2・4）
- デモの型が「僕のアプリに話しかけてみてください」になり、**相手が操作する側に回る**。見せるデモから触らせるデモへ変わり、会話が自然に生まれる（観点 3）
- スマホの文字入力問題（アイデア 1 と同じ課題）の根本解にもなる

**2026-07-06 オーナー判断: 音声対応を全体の主軸とする。** 単独機能ではなく、他アイデアを貫く横断軸として扱う。

- 音声入力は fit 判定フォームだけでなく、アイデア 3（業界別ジェネレーター）・アイデア 5（フリーチャット）の入力手段としても効く。特にフリーチャット × 音声は「本人の AI と会話する」体験そのものになる
- 音声出力（回答の読み上げ）まで組めば、スマホを相手に持たせて「話す → AI が声で返す」の完結した画が作れる。周囲が騒がしい営業現場も想定し、音声・テキストの両表示を前提にする
- 各アイデアの優先度判断は「音声と組み合わせたときに強くなるか」を加点要素にする

### 3. 「あなたの業界で AI を使うなら」ジェネレーター（相手を主役にする）

相手の業界・職種をチップで 1 つ選ぶ（または一言入れる）と、AI がその業界での活用アイデアを 2〜3 個と「黒澤ならこう関われる」を生成する。

- **なぜ効くか**: 現状のアプリは全編「黒澤の話」。この機能だけは**相手の話**になる。「お仕事なんですか？→ ちょっと入れてみてください」という営業トークの流れにそのまま乗る（観点 3 で最強）
- 生成結果が相手ごとに変わるので、「AI ってこういう使い方ができるのか」という気づきを相手自身の文脈で渡せる（観点 2）
- 出力を fit 判定と同じ構造化カードで見せれば、既存のデザイン言語をそのまま活かせる

### 4. 「AI の舞台裏」パネル（作っている人、として記憶に残す）

回答表示後にトグルで開ける小さなパネルに、実際に使ったモデル名、応答時間、送信プロンプトの要約、レスポンス JSON、概算コストなどを表示する。

- **なぜ効くか**: 「使っている人」と「作っている人」の差が一目で伝わる。相手が技術寄りなら会話が一段深くなり、非技術者にも「中身まで見せられる人」という印象が残る（観点 3・4）
- 「1 回の相談で ○ 円くらいです」はほぼ確実にウケる小ネタで、AI 導入コストの話（＝営業の本題）への橋渡しになる
- 既存のパイプライン表示の延長線上にあり、世界観が崩れない

### 5. 黒澤 AI に何でも聞けるフリーチャット（「企業 HP + AI サポート QA」の本命形）

fit 判定フォームとは別に、「黒澤って何ができる人？」「GCP は得意？」「単価は？」に AI が経歴データを根拠に答える気軽なチャット。

- **なぜ効くか**: 相談フォームは「案件を持っている人」向けで、デモ相手の多くは案件を持っていない。フリーチャットなら**誰でも遊べる**（観点 1・3)
- 「本人の隣で、本人の AI に本人のことを聞く」という画そのものが面白く、記憶に残る（観点 4）
- 回答の根拠（プロフィールのどの項目から答えたか）を小さく添えると、「AI に自分のデータを食わせて答えさせる」という仕事内容のデモにもなる

### 6. 相談結果の持ち帰り導線（デモ後に残す）

相談・生成結果を共有カード（画像または短い URL）にして、QR コードで相手のスマホに渡せるようにする。

- **なぜ効くか**: デモは見せて終わりだと消える。「さっきの結果、渡しますよ」で相手の端末にアプリの入口が残り、名刺代わりになる（観点 4）
- アイデア 3（業界別ジェネレーター）と組むと「あなた向けの AI 活用アイデア」を持ち帰らせることができ、後日の連絡の口実になる

## 次点（効くが、優先アイデアの後でよい）

- **トップページのマイクロ AI 演出**: 時間帯や閲覧文脈に応じた一言を AI が生成して表示するなど、`/ai-consult` に行く前から「このサイト自体が AI で動いている」と匂わせる小さな仕掛け。単体では弱いが、全体の世界観を底上げする
- **多言語ワンタップ切替**: 回答や問い合わせ文を英語に即変換。海外案件・外資文脈の相手には強いが、相手を選ぶ
- **相談履歴の見せ方強化**: localStorage の履歴を「これまでのデモで AI がこう答えた」ギャラリーとして見せる。デモの再現性が上がるが、初見の魅力には直結しない

## 見送りを推奨

- **本格的な RAG・ベクトル検索基盤の導入をアピール目的で行うこと**: 裏側を高度化しても画面上の体験は変わらない。デモ映えに投資する方がこのアプリの目的に合う
- **相談精度の作り込み（プロンプト・判定ロジックの精緻化)**: 営業デモでは「正しい判定」より「動いている画」が先。精度は実案件の相談が来始めてからで十分
- **アカウント・サーバー永続化などの本格 Web サービス化**: 「案件情報をサーバーに保存しない」という現在の割り切りは、むしろ営業トークで使える長所

## 推奨する順番

音声対応を主軸とする方針（2026-07-06）を反映した順番。

1. **アイデア 1（ワンタップデモ + ストリーミング）** — 既存機能の見せ方改善であり、デモの成功率に直結する。最小工数で最大効果。音声対応の前提となる「待ち時間の演出」もここで解決する
2. **アイデア 2（音声対応）** — 主軸。まず fit 判定フォームの音声入力から入り、体験の核を作る
3. **アイデア 5（フリーチャット）** — 音声と組み合わせて「本人の AI と会話する」体験に育てる。音声主軸の方針により、アイデア 3 より先に引き上げ
4. **アイデア 3（業界別ジェネレーター）** — 相手を主役にする会話のきっかけ。音声入力の適用先としても機能する
5. **アイデア 4（舞台裏パネル）と 6（持ち帰り導線）** — 印象の駄目押しとデモ後のフォロー

各アイデアの仕様・設計・実装方針は、着手判断後に `docs/tasks/backlog/` へ個別タスクとして切り出す。

## 技術メモ: 音声対応は React でできるのか（2026-07-06 追記）

結論: **できる。Flutter は不要。** 音声入出力はブラウザ標準 API として提供されており、React はそれを呼ぶだけ。フレームワークの制約ではなくブラウザ対応の問題になる。

### 音声入力（聞き取り）— 2つの実装ルート

1. **Web Speech API（SpeechRecognition）**: ブラウザ標準の音声認識で、追加コストゼロ・サーバー不要。マイクに話すとテキストが返るので、既存フォームの `inquiry` に流し込むだけで成立する。Android Chrome が最も安定、iOS Safari も 14.5 以降で対応（webkit プレフィックス付き）。Firefox は非対応だが、**このアプリは黒澤自身のスマホで見せるデモ**なので自分の端末で動けばよく、この弱点はほぼ問題にならない。日本語認識の精度も日常語彙なら実用レベル。
2. **MediaRecorder で録音 → STT API**: マイク音声を録り、Whisper 系や Google Speech-to-Text で文字起こしする方式。全ブラウザで確実に動き精度も高い代わりに、API コストとサーバー側の口が必要。ただし本プロジェクトには既に Supabase Edge Function（`consult-engineer`）で「API キーをブラウザに出さない」ゲートウェイ構造があるため、同じパターンで `transcribe` 関数を 1 本足すだけで既存アーキテクチャに素直に収まる。コストも Whisper 系なら 1 分あたり 1 円未満で、月額基準的に問題にならない規模。

### 音声出力（読み上げ）

`speechSynthesis`（ブラウザ標準 API）が全主要ブラウザで動く。日本語音声の質は端末 OS 依存（iOS はまずまず、Android は機種差あり）。「デモで聞かせて恥ずかしくない声」にしたければ TTS API の選択肢もあるが、まず標準 API で試すのが順当。

### Flutter が必要になる条件

常時待ち受け（ウェイクワード）、バックグラウンド録音、オフライン認識などネイティブ機能が要るときだけ。「画面を開いてマイクボタンを押して話す」という今回のデモ用途は Web の守備範囲に完全に収まる。

### 推奨方針

**まず Web Speech API で作って自分の端末で精度を確かめ、不満が出たら Edge Function 経由の STT API に差し替える**二段構え。差し替えても「マイクボタン → テキストが入る」という UI 側の作りは変わらないため、最初の選択で手戻りは発生しない。
