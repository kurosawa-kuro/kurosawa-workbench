# 01 要件 — 黒澤 Workbench

## サイトの位置づけ

黒澤俊文のフリーランス営業サイト。

目的は、訪問者に短時間で次を判断してもらうこと。

```text
黒澤が何者か
何を依頼できるか
自社の課題に合うか
AIに聞けば相談の入口が作れること
問い合わせに進む理由があること
```

トップ 1 枚に職務経歴・サービス説明・AI相談を詰め込まず、ページ責務を分ける。

## ページ構成

| Route | 役割 | 優先度 |
|---|---|---|
| `/` | 第一印象と分岐。`/services` と `/ai-consult` へ送る入口 | P0 |
| `/services` | 依頼できる内容。営業ページの主役 | P0 |
| `/ai-consult` | 自社課題に黒澤が合うかを AI に相談する体験ページ | P0 |
| `/career` | 経歴・技術スタック・実績背景 | P1 |
| `/contact` | 問い合わせ・業務委託相談 | P1 |
| `/cases` | 案件タイプ別の提供価値 | P2 |

`/career` を主役にしない。営業サイトとしては `/services` と `/ai-consult` が主役。

## 訪問者が感じる障壁

- この人に相談していい案件なのか分からない
- 何を依頼できるのか分からない
- 技術的に合っているのか分からない
- どこまで任せられるのか分からない
- 問い合わせ文を書くのが面倒

## ページ別要件

### `/`

分岐ハブ。次の 3 つだけを明確にする。

- 黒澤は何者か
- 何を依頼できるか
- AI に何を聞けるか

主導線:

- `/services`: 依頼できる内容を見る
- `/ai-consult`: AI に相談する
- `/career`: 経歴・実績を見る

### `/services`

依頼メニューを表示する。各メニューには以下を含める。

- 対象の困りごと
- 提供できること
- 向いている案件
- 向いていない案件
- 相談時に必要な情報

現在の依頼メニュー:

- 生成AI導入支援
- AI相談・社内ツールPoC
- AWS / Kubernetes 基盤設計
- Webアプリ開発
- 技術顧問・レビュー
- 既存システム改善

### `/ai-consult`

案件概要を自然文で入力し、黒澤に相談できそうかを AI に判定させる。

AI の回答はただのチャットではなく、次の構造で表示する。

- 回答サマリー
- 対応可能性
- 関連する経験・スキル
- 向いている依頼形態
- 追加で確認したい質問
- 問い合わせ文の下書き

現在のフォーム項目:

| フィールド | 必須 | 説明 |
|---|---|---|
| 案件概要 | 必須 | 作りたいもの・課題・技術スタック等を自然文で |
| 技術スタック | 任意 | 使いたい / 使っているスタック |
| 予算・単価感 | 任意 | 目安または未定 |
| 期限・稼働期間 | 任意 | 希望納期・参画期間 |
| 既存コードの有無 | 任意 | ゼロから / 既存あり |
| 希望する関わり方 | 任意 | 上流のみ / 実装支援 / チームリーダー / 相談しながら 等 |

相談例:

- AWSとKubernetesの運用改善を相談できますか？
- 社内向け生成AIツールのPoCを依頼できますか？
- Next.jsとGoで業務アプリを作れますか？
- 技術顧問として月数十時間の相談は可能ですか？
- 既存システムにAI検索を入れたいです

### `/career`

信用補強ページ。`app/src/data/engineer-profile.js` の内容から、職務概要、得意領域、資格・実績を表示する。

### `/cases`

案件タイプ別に価値を証明する。現在は以下 4 ケース。

- AI導入支援・PoC
- AWS / Kubernetes 基盤改善
- Webアプリ開発
- 技術顧問・設計レビュー

### `/contact`

問い合わせへの最終導線。AI相談から問い合わせ文を作って送る流れを推奨する。

直接問い合わせ時に含めてほしい情報:

- 会社名
- 氏名
- メール
- 相談内容
- 希望する支援内容
- 稼働開始時期

## AI レスポンススキーマ

```json
{
  "fit": "high | medium | low | ng",
  "canHandle": true,
  "summary": "1〜2文で対応可否と理由",
  "suggestedScope": ["ヒアリング・要件整理", "PoC 設計"],
  "risks": ["フルリモート専門のため出社不可"],
  "questions": ["既存環境はありますか？"],
  "draftInquiry": "訪問者がそのままコピーして送れる問い合わせ文"
}
```

## NG 判定ルール

Edge Function は以下を LLM 呼び出し前に判定し、該当時は `fit: ng` を返す。

| NG 条件 | 判定理由 |
|---|---|
| 常駐・出社必須・週◯出社 | フルリモートのみ対応 |
| デザイン専業・UI/UX デザイン専業 | 対象外。フロント実装は対応可 |
| DL モデル研究・論文実装専業 | 対象外。周辺インフラは対応可 |
| 薬機法対応 | 対象外 |

## 制約

| 区分 | 制約 |
|---|---|
| セキュリティ | API キーはブラウザに露出しない。Edge Functions 経由のみ |
| コスト | 入力文字数制限とブラウザ単位の日次 AI 利用回数制限を持つ |
| 耐障害性 | AI 障害時は直接問い合わせへ誘導する |
| データ | 案件情報はサーバーサイドに保存しない。相談履歴は localStorage のみ |
